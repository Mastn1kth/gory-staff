# Горы: дорожная карта проекта

Обновлено: 2026-05-30.

Эта дорожная карта не заменяет `README.md` и `docs/PROJECT_FULL_STATUS.md`. Она фиксирует порядок дальнейшей работы: что закрывать первым, зачем это нужно и как понять, что этап готов.

## Текущее состояние

Проект уже имеет рабочий MVP:

- Expo React Native APK для гостей и сотрудников.
- Node.js/Express API с PostgreSQL.
- Панель/батники для запуска на Windows-компьютере ресторана.
- Публичный адрес `https://app.gory-staff.ru` через Cloudflare relay.
- Гостевой профиль, бонусы, меню, новости, маршрут.
- Рабочая зона персонала: роли, зал, брони, меню, стоп-лист, задачи, банкеты, клиенты, аналитика.
- Offline-first кэш и очередь части рабочих действий.
- Expo push-архитектура.
- iikoCloud sync меню/стоп-листа/модификаторов, отправка гостевых заказов, pull статусов, webhook-и событий оплаты.

Главный технический долг: довести систему от MVP до надежной эксплуатации в ресторане, не расширяя хаотично функциональность.

## P0 - Зафиксировать базовую стабильность

Зачем: перед новыми функциями нужно понимать, что текущий запуск, сборка и тесты воспроизводимы.

Задачи:

- Проверить запуск через `Горы Управление.exe` и `tools\bat\START_GORY_STAFF.bat` на целевом Windows-ПК.
- Проверить, что `STOP_GORY_STAFF.bat` действительно останавливает backend, PostgreSQL-контейнер, relay, iiko connector и фоновые процессы без потери данных.
- Убрать противоречия в документации между корневыми ярлыками, `tools\bat` и панелью управления.
- Зафиксировать один актуальный путь APK: где лежит рабочий файл и как он называется.
- Прогнать `npm --workspace server test`.
- Прогнать `npm run typecheck`.
- Проверить, что `server\.env.example` соответствует фактическим обязательным env.

Готово, когда:

- сервер запускается и останавливается повторно без ручной чистки процессов;
- `/health` отвечает локально и через публичный адрес;
- README не спорит с реальной структурой файлов;
- тесты и typecheck либо проходят, либо есть честный список конкретных падений.

## P1 - Реальная проверка iiko

Зачем: iiko - самая рискованная интеграция, потому что код можно протестировать моками, но реальные payload-и и ограничения точки проявятся только на живой среде.

Задачи:

- Получить тестовую или рабочую iiko-организацию без боевых секретов в Git.
- Заполнить `server\.env`: `IIKO_ENABLED`, `IIKO_API_LOGIN`, `IIKO_ORGANIZATION_ID`, `IIKO_TERMINAL_GROUP_ID`, `IIKO_WEBHOOK_SECRET`.
- Выполнить `GET /iiko/status` и сохранить отчет по шаблону `docs/IIKO_CHECK_REPORT_TEMPLATE.md`.
- Выполнить ручной `POST /iiko/sync/menu`.
- Проверить, что категории, блюда, цены, stop-list и модификаторы попадают в PostgreSQL без дублей.
- Создать гостевой заказ и проверить отправку в iiko table order.
- Проверить `POST /iiko/sync/orders/:orderId/status` и batch status pull.
- Прогнать `tools\iiko-event-connector.js` на реальном или максимально близком JSON/JSONL событии `order-updated` и `payment-paid`.
- Проверить идемпотентность повторных событий оплаты.

Готово, когда:

- есть отчет ручной проверки без секретов;
- известно, какие payload-и реально приходят от iiko/iikoFront;
- ошибки iiko отображаются в диагностике и логах, а не теряются;
- список доработок после live-проверки добавлен в этот roadmap.

## P2 - Production security и эксплуатация

Зачем: приложение хранит гостей, сотрудников, бонусы и рабочие данные ресторана. Без базовой безопасности и backup-процедур эксплуатация рискованна.

Задачи:

- Проверить сильные `JWT_SECRET` и `GUEST_JWT_SECRET`, они должны быть разными.
- Развести dev/demo доступы и реальные доступы управляющего.
- Проверить `TRUST_PROXY`, `TRUSTED_PROXY_IPS`, CORS и публичный relay.
- Ограничить доступ к admin/iiko endpoints только нужными ролями.
- Описать backup/restore: ручной backup, ночной backup, восстановление на другом ПК.
- Проверить, что `data`, `runtime`, `backups`, `builds`, `.env` не попадают в Git.
- Добавить отдельный security checklist для передачи заказчику.
- Проверить rate limit входа и сообщения об ошибках без утечки секретов.

Готово, когда:

- можно восстановить сервер на другом ПК по инструкции;
- backup реально создается и восстанавливается;
- секреты не лежат в документации/коде;
- админские endpoints закрыты авторизацией и правами.

## P3 - Надежность данных и offline-first

Зачем: ресторанская смена не должна ломаться из-за слабого интернета или двух одновременных действий.

Задачи:

- Завершить серверную защиту конфликтов по `version` / `updated_at` для всех изменяемых сущностей.
- Проверить offline queue на разрешенных действиях: столы, брони, стоп-лист, задачи, блокнот, ожидание.
- Запретить или явно обрабатывать конфликтные offline-действия.
- Добавить видимый экран/модалку конфликтов, если сервер вернул 409.
- Проверить восстановление после перезапуска приложения с pending queue.
- Уменьшить размер mobile snapshot, если он растет сверх безопасного лимита.

Готово, когда:

- offline-действия не теряются молча;
- конфликт не перетирает чужие изменения;
- пользователь понимает, что действие сохранено, отправлено или требует ручного решения;
- есть тесты на ключевые сценарии очереди.

## P4 - Гостевые заказы и модификаторы

Зачем: iiko уже хранит модификаторы, но без UI выбора модификаторов гостевой заказ может не соответствовать реальному меню.

Задачи:

- Спроектировать UI выбора обязательных и опциональных модификаторов.
- Показать группы модификаторов в гостевом меню только там, где они нужны.
- Валидировать обязательные модификаторы на сервере перед отправкой заказа в iiko.
- Сохранять выбранные модификаторы в `guest_order_item_modifiers`.
- Отправлять модификаторы в iiko order sync в формате, который подтвердит live-проверка.
- Обновить тесты `iiko-order-sync` и гостевых маршрутов.

Готово, когда:

- блюдо с обязательным модификатором нельзя отправить без выбора;
- выбранные модификаторы видны гостю и персоналу;
- iiko получает заказ с корректными modifier items.

## P5 - Push до production

Зачем: Expo push подходит для MVP, но production-требования нужно закрыть отдельно.

Задачи:

- Решить, остается ли Expo push или нужен прямой FCM.
- Если нужен FCM: создать Firebase-проект, добавить `google-services.json` вне публичных секретов, вынести server key в env.
- Проверить получение push token на реальных Android-устройствах.
- Проверить доставку push по гостю и сотруднику.
- Добавить обработку невалидных токенов и cleanup.
- Описать эксплуатационную диагностику push.

Готово, когда:

- push работает на реальном телефоне после установки APK;
- сервер логирует доставку и ошибки;
- владелец может понять причину `no_devices`, `invalid_token` или provider error.

## P6 - APK, релизы и передача заказчику

Зачем: нужен повторяемый процесс выпуска, а не ручной набор случайных действий.

Задачи:

- Зафиксировать version/versionCode policy в `mobile\app.json`.
- Проверить локальную Android-сборку через `tools\bat\BUILD_ANDROID_APK.bat`.
- Убедиться, что APK собирается с правильным `EXPO_PUBLIC_API_URL`.
- Хранить только актуальные артефакты в `builds`, без коммита APK в Git.
- Сделать release checklist: сервер запущен, health ok, APK установлен, вход гостя, вход staff, push, offline smoke, iiko smoke.
- Описать откат на предыдущий APK/backup базы.

Готово, когда:

- новый APK можно собрать по одной инструкции;
- перед показом заказчику есть короткий smoke-test;
- понятно, как вернуться к предыдущему состоянию.

## P7 - UX и операционные доработки

Зачем: после стабилизации нужно улучшать не количество экранов, а скорость работы смены.

Задачи:

- Проверить рабочие сценарии по ролям: управляющий, хостес, официант, кухня, бар.
- Убрать лишние dev-кнопки из production-сборки или закрыть их флагом.
- Доработать экран клиентов: быстрый поиск, история, списание/начисление, связь с iiko оплатой.
- Проверить читаемость на маленьких Android-экранах.
- Проверить, что клавиатура не перекрывает формы входа/регистрации.
- Отдельно пройти гостевой flow: профиль, бонусы, новости, меню, маршрут, заказ, feedback.

Готово, когда:

- каждый role-flow проходит без тупиков;
- основные действия требуют минимум лишних переходов;
- нет экранов с техническим текстом, который видит гость или сотрудник без нужды.

## P8 - iOS и долгосрочная платформа

Зачем: iOS заявлен как будущая возможность, но требует отдельной учетной записи и процесса распространения.

Задачи:

- Получить Apple Developer аккаунт.
- Настроить bundle id `ru.gory.staff`.
- Проверить EAS iOS build.
- Проверить push permissions на iOS.
- Подготовить TestFlight или другой внутренний канал.
- Обновить документацию установки для iPhone.

Готово, когда:

- iOS build устанавливается на тестовое устройство;
- API, push и offline работают не только на Android.

## Правило приоритета

Если задача спорит с roadmap:

1. Сначала безопасность данных и запуск ресторана.
2. Потом iiko/live-интеграция.
3. Потом offline/reliability.
4. Потом UX.
5. Потом новые крупные функции.

Не добавлять новую функцию, если она ломает запуск одной кнопкой, скрыто требует секретов в Git или делает невозможным восстановление базы.
