# Горы: дорожная карта

Обновлено: 2026-05-31.

Цель roadmap - держать порядок работ. Не добавлять новые большие функции, пока не закрыты эксплуатационные риски: запуск, безопасность, iiko, данные, backup и проверка на реальных устройствах.

## Текущее состояние

MVP уже рабочий:

- Expo React Native APK для гостей и сотрудников.
- Node.js/Express API с PostgreSQL.
- Панель и батники для Windows-компьютера ресторана.
- Публичный адрес `https://app.gory-staff.ru` через Cloudflare relay.
- Гостевой профиль, бонусы, рефералы, меню, новости, маршрут.
- Рабочие разделы персонала: роли, зал, брони, меню, стоп-лист, задачи, банкеты, клиенты, аналитика.
- Offline-first кэш и часть очереди действий.
- iikoCloud sync меню/стоп-листа/модификаторов, гостевые заказы, pull статусов, события оплаты.

## Закрыто в последнем пакете

Эти пункты закрыты кодом и автотестами, но часть из них все равно требует ручной проверки на реальной среде:

- QR для списания бонусов заменен на настоящий `react-native-qrcode-svg`.
- Mobile OAuth переведен на deep link `gory-staff://oauth/{provider}`.
- Сервер OAuth валидирует mobile `redirect_uri` и использует тот же URI при обмене code.
- Twilio reservation webhook проверяет `X-Twilio-Signature`.
- Express принимает Twilio form callback через `express.urlencoded`.
- iiko staff sync подбирает уникальный `login`, если iiko отдает сотрудников с одинаковой основой логина.
- Добавлены тесты на QR/OAuth, Twilio signature и дубли login в iiko staff sync.

## P0 - Ручная приемка текущих P1

Зачем: автотесты доказывают логику, но OAuth/Twilio/iiko ломаются чаще всего на настройках внешних кабинетов и публичных URL.

Задачи:

- Зарегистрировать `gory-staff://oauth/yandex` и `gory-staff://oauth/vk` в кабинетах провайдеров.
- Проверить OAuth на реальном Android-устройстве из установленного APK.
- Если Yandex/VK не принимают custom scheme, заменить flow на server callback -> one-time ticket -> app deep link.
- Указать `TWILIO_STATUS_CALLBACK_URL` на публичный callback и проверить реальное SMS-событие от Twilio.
- Проверить, что неподписанный Twilio callback возвращает `403` на публичном домене.
- Проверить QR списания бонусов: гость показывает код, сотрудник/admin списывает бонусы, повторное списание старым кодом не проходит.
- Прогнать iiko staff sync на тестовой или рабочей организации и убедиться, что дубли логинов не останавливают sync.

Готово, когда:

- есть короткий отчет ручной проверки без секретов;
- известны реальные redirect/callback URL, которые работают в проде;
- нет P1, которые зависят только от предположений.

## P1 - Запуск и релизный контур

Зачем: ресторану нужен повторяемый запуск одной кнопкой и понятная сборка APK.

Задачи:

- Проверить `Горы Управление.exe` на целевом Windows-ПК.
- Проверить `START_GORY_STAFF.bat` и `STOP_GORY_STAFF.bat` без ручной чистки процессов.
- Зафиксировать один актуальный путь APK: `builds\Gory-latest.apk`.
- Проверить `/health` локально и через `https://app.gory-staff.ru`.
- Убедиться, что `server\.env.example` покрывает обязательные env.
- Описать release smoke checklist: сервер, health, APK, гостевой вход, staff вход, QR бонусов, OAuth, Twilio, iiko smoke.

Готово, когда:

- сервер стартует/останавливается повторно;
- APK собирается и устанавливается;
- есть понятный smoke checklist перед показом заказчику.

## P2 - Security и эксплуатация

Зачем: проект хранит гостей, сотрудников, бонусы и операционные данные ресторана.

Задачи:

- Проверить сильные и разные `JWT_SECRET` / `GUEST_JWT_SECRET`.
- Развести demo/dev доступы и реальные доступы управляющего.
- Проверить `TRUST_PROXY`, `TRUSTED_PROXY_IPS`, CORS и публичный relay.
- Закрыть admin/iiko endpoints авторизацией и правами.
- Описать backup/restore: ручной backup, ночной backup, восстановление на другом ПК.
- Проверить, что `data`, `runtime`, `backups`, `builds`, `.env`, APK и временные отчеты не попадают в Git.
- Добавить security checklist перед передачей проекта.

Готово, когда:

- сервер можно восстановить на другом ПК по инструкции;
- backup реально создается и восстанавливается;
- секреты не лежат в коде или документации;
- админские endpoints закрыты ролями и правами.

## P3 - Live-проверка iiko

Зачем: iiko нельзя надежно закрыть моками. Реальные payload-и и ограничения точки проявляются только на живой среде.

Задачи:

- Заполнить `server\.env` для тестовой или рабочей iiko-организации.
- Выполнить `GET /iiko/status`.
- Выполнить ручной `POST /iiko/sync/menu`.
- Проверить категории, блюда, цены, stop-list, модификаторы без дублей.
- Создать гостевой заказ и проверить отправку в iiko table order.
- Проверить pull статусов заказов.
- Прогнать `tools\iiko-event-connector.js` на событиях `order-updated` и `payment-paid`.
- Проверить идемпотентность повторных событий оплаты.
- Проверить staff sync, включая одинаковые ФИО/login.

Готово, когда:

- есть отчет ручной проверки без секретов;
- ошибки iiko видны в диагностике и логах;
- список реальных доработок после live-проверки добавлен в roadmap.

## P4 - Надежность данных и offline-first

Зачем: смена не должна ломаться из-за слабого интернета или двух одновременных действий.

Задачи:

- Завершить серверную защиту конфликтов по `version` / `updated_at`.
- Проверить offline queue для столов, броней, стоп-листа, задач, блокнота и листа ожидания.
- Запретить или явно обработать конфликтные offline-действия.
- Показать пользователю конфликт, если сервер вернул `409`.
- Проверить восстановление pending queue после перезапуска приложения.
- Следить за размером mobile snapshot.

Готово, когда:

- offline-действия не теряются молча;
- конфликт не перетирает чужие изменения;
- есть тесты на ключевые сценарии очереди.

## P5 - Гостевые заказы и модификаторы

Зачем: без UI выбора модификаторов гостевой заказ может не соответствовать реальному меню iiko.

Задачи:

- Спроектировать UI выбора обязательных и опциональных модификаторов.
- Показывать группы модификаторов только там, где они нужны.
- Валидировать обязательные модификаторы на сервере.
- Сохранять выбранные модификаторы в `guest_order_item_modifiers`.
- Отправлять модификаторы в iiko в формате, подтвержденном live-проверкой.
- Обновить тесты `iiko-order-sync` и гостевых маршрутов.

Готово, когда:

- блюдо с обязательным модификатором нельзя отправить без выбора;
- модификаторы видны гостю и персоналу;
- iiko принимает заказ с modifier items.

## P6 - UX и операционные доработки

Зачем: после стабилизации нужно ускорять работу смены, а не просто добавлять экраны.

Задачи:

- Пройти role-flow: управляющий, хостес, официант, кухня, бар.
- Убрать dev-кнопки из production или закрыть флагом.
- Проверить читаемость на маленьких Android-экранах.
- Проверить формы входа/регистрации с открытой клавиатурой.
- Отдельно пройти гостевой flow: профиль, бонусы, OAuth, QR, новости, меню, маршрут, заказ, feedback.

Готово, когда:

- основные действия требуют минимум лишних переходов;
- нет технического текста, который видит гость без нужды;
- role-flow проходит без тупиков.
