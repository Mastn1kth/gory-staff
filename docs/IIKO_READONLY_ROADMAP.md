# iikoCloud read-only roadmap

Статус на 2026-05-29: документ исторический. Проект уже вышел за рамки read-only: кроме импорта меню/stop-list добавлена отправка гостевых заказов в iiko через table orders API. Актуальный общий статус см. в `README.md` и `docs/PROJECT_FULL_STATUS.md`.

Этот документ фиксирует структуру дальнейших работ по iikoCloud без перехода к заказам, оплате, кассе и автоматической синхронизации.

## Текущее состояние

Уже есть read-only этап:

- `GET /iiko/status` для диагностики;
- `POST /iiko/sync/menu` для ручного sync меню;
- импорт категорий в `menu_categories`;
- импорт позиций меню и цен в `menu_items`;
- read-only обработка stop-list в `stop_list`;
- журнал sync в `iiko_sync_log`;
- ручной чек-лист `docs/IIKO_MANUAL_CHECKLIST.md`.

Интеграция не создает заказы, не принимает оплату, не работает с кассой, не делает фискализацию и не отправляет данные на кухню.

Отдельно добавлен безопасный прием факта уже прошедшей оплаты от локального iikoFront/edge-коннектора: `POST /iiko/events/payment-paid`. Это не проведение оплаты и не кассовая операция. Endpoint нужен, чтобы после закрытия/оплаты заказа в iiko приложение сопоставило гостя и отправило push `Оцените визит`.

## Что наращиваем до реальной проверки

Цель ближайшего этапа - подготовить проект так, чтобы реальная проверка прошла по понятной схеме и оставила воспроизводимый обезличенный результат.

Без реального iikoCloud сейчас можно нарастить:

- структуру документации;
- шаблон отчета ручной проверки;
- troubleshooting по типовым ошибкам;
- список безопасных артефактов, которые можно сохранять после проверки;
- правила, какие данные нельзя публиковать;
- будущие unit-тесты на fake iiko responses, если потребуется расширять покрытие без обращения к iikoCloud.

## Этап 1. Документация перед проверкой

Статус: текущий этап.

Задачи:

- держать `docs/IIKO_MANUAL_CHECKLIST.md` как основную пошаговую инструкцию;
- держать `docs/IIKO_CHECK_REPORT_TEMPLATE.md` как шаблон результата проверки;
- держать `docs/IIKO_TROUBLESHOOTING.md` как место для ошибок и решений;
- не добавлять реальные iiko credentials в документы;
- не описывать неподтвержденные функции как реализованные.

Критерий готовности:

- проверяющий понимает, какие env нужны;
- проверяющий понимает, как получить staff token;
- проверяющий понимает, какие endpoint вызвать;
- проверяющий понимает, какие SQL-запросы выполнить;
- проверяющий знает, что именно нельзя коммитить.

## Этап 2. Локальное структурное усиление без iikoCloud

Этот этап можно делать без доступа к тестовой организации.

Допустимые задачи:

- добавить больше unit-тестов на `syncIikoMenu` с fake nomenclature и fake stop-list;
- проверить сценарии пустого меню на fake response;
- проверить сценарии stop-list на fake response;
- проверить, что полный `IIKO_API_LOGIN` не попадает в `/iiko/status`;
- проверить, что disabled sync не обращается к iiko client;
- уточнять docs по фактическим тестам.

Запрещено на этом этапе:

- делать реальные запросы к iikoCloud;
- добавлять cron/scheduler;
- добавлять background jobs;
- добавлять создание заказов;
- добавлять оплату;
- считать payment-paid webhook проведением оплаты;
- добавлять кассу;
- добавлять фискализацию;
- добавлять отправку на кухню;
- добавлять новый UI.

## Этап 3. Первая ручная проверка на тестовой iiko-организации

Этот этап начинается только когда есть безопасные тестовые credentials.

Проверяется:

- `/iiko/status`;
- ручной `/iiko/sync/menu`;
- `iiko_sync_log`;
- категории;
- блюда;
- цены;
- stop-list, если в тестовой организации есть stop-позиции;
- архивирование, только если есть безопасная тестовая позиция.

Результат заносится в `docs/IIKO_CHECK_REPORT_TEMPLATE.md` как копия отчета без секретов.

## Этап 4. Уточнение по результатам проверки

После первой реальной проверки можно делать только read-only улучшения:

- дополнить troubleshooting реальными обезличенными ошибками;
- уточнить чек-лист;
- добавить unit-тесты под найденные edge cases;
- улучшить диагностику `/iiko/status`, если текущих полей недостаточно;
- отдельно спланировать read-only stop-list, если проверка покажет пробелы.

## Что не планируется в этой ветке

В рамках этой read-only ветки не планируется:

- создание заказов в iiko;
- изменение заказов;
- оплата;
- касса;
- фискализация;
- кухня;
- автоматический sync;
- cron/scheduler;
- production automation.

Эти темы можно планировать только после подтвержденного read-only меню и отдельного решения по рискам.
