# Дизайн ролей и прав доступа

Дата: 2026-05-22

## Контекст

В приложении уже есть рабочая модель ролей, но она слишком грубая:

- `manager` сейчас имеет полный доступ через `*`.
- `administrator` почти повторяет управляющего, но без полного системного доступа.
- `kitchen` объединяет шеф-повара и обычного повара.
- `technical` сейчас больше похож на охрану/техперсонал, а не на системного техника.

Новая модель должна разделить бизнес-доступ, рабочие роли ресторана и технический супердоступ. Сейчас делаем фиксированную матрицу ролей, но называем права так, чтобы потом можно было расширить систему до точечных разрешений без полной переделки.

## Выбранный подход

Выбран вариант 3: фиксированная матрица ролей сейчас, расширяемая модель прав внутри.

Причина:

- ресторану сейчас нужна понятная рабочая схема, а не сложный конструктор прав;
- меньше риск случайно открыть сотруднику лишние разделы;
- при этом права остаются отдельными строками, чтобы позже можно было точечно добавлять или убирать возможности.

## Роли

### `technician` / Техник системы

Назначение: человек, который чинит и обслуживает приложение.

Первая вкладка: `Админ / техчасть`.

Доступ: всё.

Может:

- видеть все вкладки;
- менять меню, зал, клиентов, сотрудников, роли, графики, задачи, банкеты;
- пользоваться аналитикой;
- открывать техническую часть, диагностику, системные проверки и служебные настройки.

Не ограничиваем ресторанными правилами, потому что это роль полного системного обслуживания.

### `owner` / Владелец

Назначение: собственник ресторана, которому важны деньги, клиенты и контроль людей, а не ежедневная кухня.

Первая вкладка: `Аналитика / прибыль`.

Вкладки:

- аналитика / прибыль;
- клиенты;
- персонал;
- график;
- пульс;
- уведомления;
- профиль.

Может:

- смотреть аналитику, прибыль и итоги;
- смотреть и вести клиентскую базу;
- смотреть персонал;
- назначать роли сотрудникам;
- видеть график и общую картину ресторана.

Не должен видеть:

- отдельную вкладку меню;
- стоп-лист как рабочий экран;
- кухонные заявки и кухонную операционку;
- рабочие банкеты кухни;
- техническую часть системы.

### `manager` / Управляющий

Назначение: управляет рестораном и сменой на уровне операций, но не обслуживает техсистему.

Первая вкладка: `Пульс`.

Приоритет вкладок:

1. пульс;
2. план зала;
3. персонал;
4. клиенты;
5. аналитика.

Остальные вкладки:

- брони;
- лист ожидания;
- меню;
- стоп-лист;
- график;
- банкеты;
- заявки;
- уведомления;
- профиль;
- объявления;
- правила;
- чат;
- админ-раздел ресторана.

Может:

- управлять ресторанными операциями;
- менять ресторанные данные;
- работать с персоналом;
- работать с клиентами;
- смотреть аналитику;
- управлять меню и стоп-листом;
- назначать роли обычным сотрудникам.

Не должен видеть:

- техническую часть системы;
- серверные проверки;
- логи;
- сборки;
- системную диагностику.

### `administrator` / Администратор смены

Назначение: управляет сменой в зале, посадкой, бронями, задачами и банкетами.

Первая вкладка: `Пульс`.

Вторая вкладка: `План зала`.

Вкладки:

- пульс;
- план зала;
- брони;
- лист ожидания;
- график / персонал на смене;
- банкеты;
- задачи;
- уведомления;
- профиль.

Может:

- управлять планом зала в рамках смены;
- работать с бронями и листом ожидания;
- смотреть сотрудников на смене и график;
- вести банкетные задачи;
- ставить и закрывать задачи.

Не должен видеть:

- клиентскую базу;
- аналитику прибыли;
- назначение ролей;
- техническую часть;
- редактирование меню.

### `hostess` / Хостес

Назначение: встреча гостей, посадка, брони и очередь.

Первая вкладка: `План зала`.

Вкладки:

- план зала;
- брони;
- лист ожидания;
- уведомления;
- профиль.

Может:

- работать с посадкой;
- менять статусы столов в рамках своей работы;
- создавать и обновлять брони;
- вести лист ожидания.

Не должен видеть:

- клиентов как базу CRM;
- персонал;
- аналитику;
- меню;
- задачи;
- админку.

### `waiter` / Официант

Назначение: рабочий экран официанта.

Первая вкладка: `Блокнот`.

Вкладки:

- блокнот;
- меню;
- мои столы;
- уведомления;
- профиль.

Может:

- вести свой блокнот;
- смотреть меню;
- работать со своими столами;
- получать уведомления.

Не должен видеть:

- план зала как общий управленческий экран;
- брони;
- клиентов;
- персонал;
- аналитику;
- редактирование меню.

### `chef` / Шеф-повар

Назначение: главный по кухне и меню.

Первая вкладка: `Меню`.

Вкладки:

- меню;
- стоп-лист;
- заявки;
- задачи;
- банкеты;
- уведомления;
- профиль.

Может:

- полностью управлять меню;
- менять категории;
- менять блюда;
- менять состав, фото, цены и описание;
- менять наличие;
- управлять стоп-листом;
- работать с кухонными заявками, задачами и банкетами.

Не должен видеть:

- клиентов;
- персонал;
- роли;
- аналитику прибыли как управленческий раздел;
- техническую часть.

### `cook` / Повар

Назначение: рабочий кухонный сотрудник.

Первая вкладка: `Стоп-лист`.

Вкладки:

- стоп-лист;
- задачи;
- заявки;
- банкеты;
- уведомления;
- профиль.

Может:

- работать со стоп-листом;
- видеть и выполнять кухонные задачи;
- создавать и обновлять кухонные заявки;
- видеть банкетную кухонную работу.

Не должен видеть:

- редактирование меню;
- цены;
- категории;
- состав и фото блюд как управление меню;
- сотрудников;
- клиентов;
- роли;
- аналитику.

### `bar` / Бармен

Назначение: рабочий барный сотрудник.

Первая вкладка: `Меню`.

Вкладки:

- меню;
- стоп-лист;
- заявки;
- задачи;
- банкеты;
- уведомления;
- профиль.

Может:

- смотреть меню;
- работать с барной частью меню;
- менять наличие напитков и барных позиций;
- работать с барным стоп-листом;
- создавать заявки по бару;
- видеть банкетные позиции по бару.

Не должен видеть:

- редактирование всего меню;
- цены как управленческую настройку;
- сотрудников;
- клиентов;
- роли;
- аналитику.

### `technical_staff` / Техперсонал

Назначение: охрана, уборка, ремонт, свет, звук и похожие рабочие роли.

Первая вкладка: `Задачи`.

Вкладки:

- задачи;
- график;
- уведомления;
- профиль.

Может:

- видеть свои задачи;
- закрывать задачи;
- смотреть график;
- получать уведомления.

Не должен видеть:

- меню;
- кухню;
- клиентов;
- персонал;
- роли;
- аналитику;
- зал;
- брони;
- техническую часть системы.

### `pending` / Новый сотрудник

Назначение: сотрудник зарегистрировался, но ему еще не назначили роль.

Первая вкладка: `Профиль`.

Вкладки:

- профиль;
- экран ожидания назначения роли.

Может:

- видеть свой профиль;
- понимать, что доступ появится после назначения роли.

Не должен видеть рабочие разделы.

## Разделение похожих ролей

`technician` и `technical_staff` обязательно должны быть разными ролями.

`technician` - это системный техник с полным доступом.

`technical_staff` - это обычный техперсонал ресторана, которому нужны только задачи, график, уведомления и профиль.

`chef` и `cook` тоже обязательно должны быть разными ролями.

`chef` управляет меню полностью.

`cook` работает только с кухонной операционкой и не редактирует меню.

## Миграция текущих ролей

Текущие роли нужно привести к новой схеме:

- существующий `manager` больше не должен иметь технический `*`;
- создать `technician` как единственную роль полного системного доступа;
- добавить `owner`;
- разделить текущий `kitchen` на `chef` и `cook`;
- заменить смысл текущего `technical` на `technical_staff` или добавить новую роль `technical_staff` и мигрировать туда охрану/ремонт/свет/звук;
- оставить `administrator`, `hostess`, `waiter`, `bar`, `pending`.

Для старых пользователей:

- текущий главный пользователь из `INITIAL_MANAGER_LOGIN` должен стать `manager`, если это управляющий, или `owner`, если это владелец бизнеса;
- демо-пользователь `u-kitchen` с позицией `Шеф-повар` должен стать `chef`;
- демо-пользователь `u-cook` с позицией `Повар` должен стать `cook`;
- демо-пользователь `u-security` должен стать `technical_staff`.

## Поведение интерфейса

После входа сервер должен возвращать:

- роль пользователя;
- список прав;
- список доступных вкладок;
- первую вкладку для этой роли.

Клиент не должен сам угадывать, какие разделы доступны. Он может иметь локальный порядок вкладок для удобства, но окончательный список должен идти от сервера.

Если пользователь пытается открыть недоступный раздел, интерфейс должен показать короткое сообщение без технических деталей: `Раздел недоступен для вашей роли.`

## Поведение сервера

Сервер должен проверять права на каждом действии, а не только скрывать вкладки в приложении.

Правило:

- скрытая вкладка в приложении - это удобство;
- серверная проверка права - это защита.

Если права нет, сервер возвращает `403`.

Сообщения должны быть понятными:

- `Раздел доступен только управляющему, владельцу или технику.`
- `Меню может менять только шеф-повар, управляющий или техник.`
- `Технический раздел доступен только технику системы.`

## Тестирование

Нужно проверить:

- каждая роль получает правильный список вкладок;
- первая вкладка совпадает с согласованной таблицей;
- `owner` не видит меню, стоп-лист и техчасть;
- `manager` не видит техчасть;
- `technician` видит всё;
- `chef` может менять меню;
- `cook` не может менять меню;
- `waiter` начинает с блокнота;
- `technical_staff` начинает с задач;
- сервер возвращает `403` при попытке вызвать чужой маршрут напрямую;
- старые демо-пользователи после миграции получают правильные роли.
