# iikoCloud troubleshooting

Этот документ нужен для накопления типовых ошибок read-only iiko-интеграции. Реальные ключи, полный `IIKO_API_LOGIN`, пароли, bearer token и production-данные сюда не добавлять.

## Базовый порядок диагностики

1. Проверить `GET /iiko/status`.
2. Проверить `env.ok` и `env.missing`.
3. Проверить последнюю строку `iiko_sync_log`.
4. Проверить server logs.
5. Проверить, что `server\.env` заполнен, но не копировать реальные значения в отчет.
6. Проверить, что используется тестовая iikoCloud-организация.

## `enabled = false`

Возможные причины:

- `IIKO_ENABLED` не равен `true`;
- `IIKO_API_LOGIN` пустой.

Что делать:

- проверить `server\.env`;
- перезапустить сервер;
- снова открыть `/iiko/status`.

## `env.ok = false`

Текущий код считает обязательными:

- `IIKO_ENABLED`;
- `IIKO_API_LOGIN`;
- `IIKO_ORGANIZATION_ID`.

Что делать:

- посмотреть `env.missing`;
- заполнить отсутствующие значения в `server\.env`;
- не вставлять реальные значения в документацию или чат;
- перезапустить сервер.

## `POST /iiko/sync/menu` вернул `status = "disabled"`

Это не ошибка iikoCloud. Это значит, что текущая конфигурация выключила sync.

Что делать:

- проверить `IIKO_ENABLED`;
- проверить, что `IIKO_API_LOGIN` не пустой;
- проверить `/iiko/status`.

## `POST /iiko/sync/menu` вернул HTTP `502`

Это значит, что sync завершился `failed`.

Что смотреть:

- `error` в ответе endpoint;
- `lastSync.error` в `/iiko/status`;
- `iiko_sync_log.error_message`;
- server logs.

Частые причины:

- неверный `IIKO_API_LOGIN`;
- неверный `IIKO_ORGANIZATION_ID`;
- iikoCloud недоступен;
- сеть недоступна;
- ошибка схемы локальной БД;
- неожиданный формат ответа iikoCloud.

## Пустые категории или позиции

Возможные причины:

- в тестовой организации действительно пустое меню;
- organization id указывает не на ту организацию;
- в iikoCloud данные скрыты, удалены или не включены в меню;
- ответ nomenclature не содержит видимых groups/products.

Что делать:

- не считать это багом без сверки с тестовой организацией;
- проверить organization id с владельцем тестовой iikoCloud;
- зафиксировать обезличенный результат в отчете.

## Stop-list пустой

Пустой `stop_list` не всегда ошибка.

Возможные причины:

- в тестовой iikoCloud нет stop-позиций;
- `IIKO_TERMINAL_GROUP_ID` задан не для той terminal group;
- stop-list есть, но относится к другой terminal group.

Что делать:

- проверить, есть ли stop-позиции в тестовой iikoCloud;
- проверить terminal group id;
- если stop-list в iiko пустой, записать "не применимо" в отчете.

## Полный `IIKO_API_LOGIN` виден в ответе или логах

Это критичная проблема безопасности.

Что делать:

- не публиковать такой ответ;
- удалить секрет из отчета, скриншота или лога перед передачей;
- отдельно завести задачу на исправление маскирования;
- проверить `/iiko/status` после исправления.

## Ошибка прав доступа

Если endpoint возвращает `401` или `403`:

- нет bearer token;
- token истек;
- пользователь не найден;
- у роли нет `manage:menu`.

Что делать:

- заново получить token через `/auth/login`;
- использовать staff-пользователя с правом `manage:menu`;
- не сохранять token в файлы проекта.

## Что пока не диагностируется

На текущем этапе не реализовано / не проверяется:

- создание заказов;
- оплата;
- касса;
- фискализация;
- кухня;
- UI выбора модификаторов в заказе;
- автоматический menu sync.
